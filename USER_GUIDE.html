1. What is Maqueen Lab?

Not a "robot remote" — a component lab. Each actuator and sensor on the Maqueen Lite v4 has dedicated controls so you can manipulate it live, see the wire-format command being sent, and watch sensor data react. Everything runs in the browser via Web Bluetooth; no install, no app store, just an HTTPS page.

The app forks bit-playground v1.2.0 and adds a Maqueen-specific layer:

  • A 🤖 Maqueen tab with native Drive / Servos / Lights / NeoPixels / Buzzer panels
  • A live sensor strip across the top showing line sensors, ultrasonic distance, IR codes, accelerometer, and BLE round-trip stats
  • A BLE scheduler that adds sequence numbers and echo-confirmed round-trips on top of the existing UART layer
  • A Mechanic Kit picker for Forklift / Loader / Beetle / Push attachments
  • All bit-playground tabs (Sensors / Motors / GamePad / Graph / 3D) work alongside the Maqueen tab via firmware bridge verbs

3. First flash — firmware setup

  1. Open makecode.microbit.org, create a new project.
  2. Click Extensions and add pxt-maqueen:
    • Search for maqueen, OR
    • Paste the URL https://github.com/DFRobot/pxt-maqueen
  3. Switch the editor to JavaScript mode (top right).
  4. In Maqueen Lab, click the 🤖 Firmware button in the Connect card → Copy Code.
  5. Paste it into MakeCode (Ctrl+A then Ctrl+V to replace everything).
  6. Edit the build stamp at the top if you want a custom version label: 
    const BUILD_VERSION = "0.1.1"
const BUILD_DATE = "2026-04-26"
  7. Click Download to get the .hex file. Drag it onto your micro:bit (it appears as a USB drive).
  8. The micro:bit reboots, runs the firmware, and shows a "✗" icon (no BLE connection yet).

NeoPixel + Bluetooth conflict (current limitation): the standard neopixel extension and bluetooth extension can't coexist in MakeCode by default — adding NeoPixel triggers "Cannot add neopixel extension. Remove all the blocks from the bluetooth extension and try again."

Three workarounds:

  • Skip RGB for now — comment out the NeoPixel block in the firmware (everything else will work). The web app's RGB: commands will be no-ops on the robot but won't error.
  • Try a BLE-safe NeoPixel fork like Bohwaz/pxt-neopixel or openblockcc/pxt-neopixel-ble (paste the URL in the Extensions search).
  • If you have a working NeoPixel sketch already, share the neopixel.create(...) line with the project so we can replicate the working setup.

USB serial boot banner

If you open a serial terminal at 115200 baud over USB, the firmware logs a boot banner and every received command:

=========================================================
[boot] Maqueen Lab firmware v0.1.1 built 2026-04-26
[boot] hardware: Maqueen Lite v4 (ROB0148)
[boot] BLE UART ready — waiting for connection
=========================================================

4. Connect via BLE

  1. Power on the Maqueen Lite v4 (slide switch on the underside).
  2. Open the Maqueen Lab page in Chrome / Edge.
  3. Click Connect in the Connect card (top of page).
  4. In the browser pairing dialog, pick the device named BBC micro:bit [xxxxx] or uBit [xxxxx].
  5. The status chip turns green ("Connected"), the micro:bit displays a "✓" icon, and the live sensor strip starts updating.

The app remembers your last device and will offer to reconnect on next visit. It also reconnects automatically if the BLE link drops briefly.

8. BLE UART protocol — verb reference

Every command carries a sequence number (#N). Micro:bit replies ECHO:N <verb> on receipt. The web app validates round-trip and surfaces lost packets / latency in the live sensor strip.

Commands (browser → micro:bit)

VerbArgsEffect
M:L,Rsigned -255..255 eachDrive motors (M1=left, M2=right)
STOP—Brake both motors
LED:i,si ∈ 0,1; s ∈ 0,1Simple LED L/R on/off
RGB:i,r,g,bi ∈ 0..3 or *; r,g,b ∈ 0..255Set NeoPixel color
SRV:i,ai ∈ 1,2; a ∈ 0..180Set servo angle
BUZZ:f,msf Hz (0=off), ms durationPlay tone
LINE?—Reply LINE:l,r
DIST?—Reply DIST:cm
IR?—Reply IR:code

9. Hardware reference

See pinout.html for the full pin map with line-by-line evidence from the pxt-maqueen source. Quick reference:

PinUsed by
P0Buzzer (default music pin) — also Gravity port
P1 / P2Ultrasonic TRIG / ECHO — also Gravity ports
P8 / P12Simple LEDs Left / Right (digital)
P13 / P14Line sensors Left / Right (digital)
P15NeoPixel data line (4 pixels) — assumed default
P19 / P20I2C SCL / SDA — motor driver, servos, future I2C add-ons

I2C 0x10 register map: 0x00=M1, 0x02=M2, 0x14=S1, 0x15=S2.

Conflict warnings: Gravity ports P0/P1/P2 share pins with the buzzer and ultrasonic. Plugging a Gravity sensor into one disables the on-board function on that pin.

10. Troubleshooting

SymptomLikely causeFix
Connect dialog shows no devices Bluetooth off, micro:bit not advertising, or robot not powered Power-cycle the Maqueen, re-flash the firmware, ensure system Bluetooth is on
Connects then immediately drops Battery low, or laptop too far from robot Replace AAAs / charge the lithium cell, move within 1 m for first pair
"echo timeout" warnings in console BLE link dropping packets, robot not connected, or stale sequence buffer Check connection status; reduce sweep speed; move closer; replace batteries
UNKNOWN_VERB reply App sent a verb the firmware doesn't recognize (version mismatch) Re-flash the latest firmware (Firmware modal → Copy Code → MakeCode)
"Cannot add neopixel extension" MakeCode blocks NeoPixel + Bluetooth coexistence by default See the warning in §3 — try a BLE-safe NeoPixel fork or skip RGB for now
Slider moves but motor doesn't Servo calibration too narrow, kit physically blocking, or speed below motor's stiction Manually free the servo arm, raise speed slider above ~80, check kit alignment
Only one of two LEDs blinks (Fixed in v0.1.0+) Was a rate-limit issue in the scheduler Already fixed — refresh the page
Page looks broken / old version stuck Service worker cached old assets Click 🗑️ Reset in the header, or hard-reload (Ctrl+Shift+R)

Architecture rules

  • js/ble.js is sacred — never modified. Its API surface (sendLine, handleUartLine, reconnect logic, writeChar) is the working BLE foundation. New layers wrap it via window.handleUartLine monkey-patch (see js/ble-scheduler.js for the pattern).
  • Firmware bridge verbs — to keep bit-playground tabs working, add new verbs to firmware that translate to existing logic. See SERVO1: → SRV:1, alias for example.
  • Coalesce vs await — for fast slider drags, send with {coalesce: true} (latest-wins). For paired commands (LED L+R, two pixels), await send(...) sequentially so each gets through the rate limit.

Firmware version bump workflow

# Edit firmware/v1-maqueen-lib.ts
python tools/bump-firmware.py        # 0.1.1 → 0.1.2 + today's date + re-embed
git add -A
git commit -m "fix: ..."
git push                              # GitHub Pages redeploys in ~1 min

Flags: --bump major|minor|patch (default: patch), --no-bump (just refresh date).

Adding a new firmware verb

  1. Add the verb to the dispatch in firmware/v1-maqueen-lib.ts (inside the big if/else in onUartDataReceived).
  2. Document it in the file header comment block.
  3. Run python tools/bump-firmware.py.
  4. Add the corresponding UI control in js/maqueen-tab.js, sending via window.bleScheduler.send('VERB:args').
  5. If it's a query verb, parse the reply in js/maqueen-panel.js's onReply handler.

Local development

cd maqueen-app
python -m http.server 9130            # or: npx serve .
# Open http://localhost:9130

## 1. Vision

Not a "robot remote" — a **component lab**. Every actuator and sensor on the Maqueen Lite gets a dedicated **Explorer** page where kids can:

- See the component visualized photorealistically.
- Manipulate it via sliders, buttons, color pickers.
- Read live the code that drives it (TypeScript using `maqueen.*` lib first, raw `pins.*` later).
- Run guided micro-challenges ("Make the arm wave", "Find the center").
- Watch sensor data react to commands on a shared telemetry graph.

All control flows over BLE UART. Every command is echo-confirmed.

## 2. Hardware reference — Maqueen Lite v4

### Pin map

| Pin | Used by | Direction | Notes |
|---|---|---|---|
| **P0** | Buzzer | Out (PWM) | Default `music` pin; also Gravity port |
| **P1** | Ultrasonic TRIG | Out | Also Gravity port — conflict if reassigned |
| **P2** | Ultrasonic ECHO | In | Also Gravity port — conflict if reassigned |
| P3, P4, P6, P7, P9, P10 | LED matrix cols | — | Free if display disabled |
| P5 | Button A | In | micro:bit built-in |
| **P8** | LED Left (simple) | Out | Digital ON/OFF |
| P11 | Button B | In | micro:bit built-in |
| **P12** | LED Right (simple) | Out | Digital ON/OFF |
| **P13** | Line sensor Left | In | Digital 0/1 |
| **P14** | Line sensor Right | In | Digital 0/1 |
| P15, P16 | — | free | — |
| **P19** | I2C SCL | — | Motor + RGB + Servos (addr **0x10**) |
| **P20** | I2C SDA | — | Same I2C bus |

### I2C 0x10 register map (motor driver chip)

| Register | Function |
|---|---|
| `0x00` | Motor M1 (left): direction + speed |
| `0x02` | Motor M2 (right): direction + speed |
| `0x14` | Servo S1 angle |
| `0x15` | Servo S2 angle |
| `0x32` | 4× RGB LED data (16M colors each) |

### Free for kid add-ons

P3, P4, P6, P7, P9, P10, P15, P16 (P3/4/6/7/9/10 share LED display — disable display first).

### Hardware components

- 2× N20 motors (I2C-driven)
- 4× RGB ambient LEDs (I2C reg 0x32)
- 2× simple LEDs (P8, P12)
- 2× IR digital line sensors (P13, P14)
- 1× SR04 ultrasonic (P1 trig, P2 echo)
- 1× IR receiver (NEC, internal pin)
- 1× buzzer (P0)
- 2× servo ports S1, S2 (I2C 0x14, 0x15)
- 3× Gravity extension ports (P0, P1, P2 — all conflict-prone)
- I2C bus (P19/P20) for add-ons

## 3. Architecture decisions

### 3.1 BLE protocol — sequence + echo confirmation

Every command carries a sequence number. Micro:bit echoes it. Web app validates round-trip.

**Wire format:**
```
Web → bit:   #42 SRV:1,90
bit → Web:   ECHO:42 SRV:1,90
bit → Web:   DONE:42                (optional, for slow ops)
```

**Web side (`js/ble-scheduler.js`):**
- `pendingCmds: Map<seq, {verb, ts, resolve, reject}>`
- 500 ms timeout → reject + UI red flash + console log
- Bench panel: sent / echoed / lost / avg latency
- Animation streams: coalesce *before* assigning seq

**UI per command (in code panel):**
```
ble.send('SRV:1,90')   ✅ 47ms
ble.send('SRV:1,95')   ⏳
ble.send('SRV:1,100')  ❌ timeout
```

### 3.2 USB serial dual logging

Firmware mirrors everything to USB serial at 115200 baud:

```
[boot] MaqueenApp firmware v0.1.0 built 2026-04-26 01:23 UTC
[boot] hardware: Maqueen Lite v4
[boot] BLE name: BBC micro:bit [vagiv]
[ble]  connected
[rx]   #42 SRV:1,90
[exec] maqueen.servoRun(S1, 90)
[tx]   ECHO:42 SRV:1,90
```

**Verbosity** via BLE: `LOG:0` (silent), `LOG:1` (rx/tx), `LOG:2` (+exec), `LOG:3` (+sensor polls).

### 3.3 Build banner

`BUILD_VERSION` and `BUILD_DATE` are two `const` lines at top of firmware. Edit manually before each flash. Banner prints at boot.

### 3.4 Web-driven animations

Micro:bit firmware is **dumb**: no on-board loops. All animations (sweep, blink, rainbow) generated by the web app sending streamed commands.

**BLE throughput cap:** ~8 cmd/sec reliably. Scheduler:
- Coalesces fast slider drags to latest value.
- Rate-limits per characteristic.
- Animation registry: `scheduler.animate(verb, frame → value, fps)` synced to BLE rate, not 60 Hz.

**Visual stays at 60 fps locally** — the kid never feels lag, even when BLE confirms slower.

### 3.5 Two firmware tracks (educational progression)

```
firmware/
├── v1-maqueen-lib.ts        ← uses maqueen.*  (default, ships v0.1.0)
└── v2-raw-pins.ts           ← uses pins.*    (advanced, ships later)
```

Same UART verbs in both → same web app works with either. Code panel shows both translations side-by-side per action.

## 4. Component Explorer pattern

Every component uses the same layout:

```
┌─────────────────────────────────────┐
│  Visual (live, animated, 60fps)     │  ~40%
├─────────────────────────────────────┤
│  Controls (sliders, buttons)        │  ~30%
├─────────────────────────────────────┤
│  Live code (TS using maqueen lib) 📋│  ~20%
├─────────────────────────────────────┤
│  Try this: [Preset A] [B] [C]       │  ~10%
└─────────────────────────────────────┘
```

**First contact**: 2-second auto-demo plays the first time each Explorer opens.

**Code panel** shows two versions side-by-side: `maqueen.*` (lib) and `pins.*` (raw). Kids see the abstraction, then the mechanism.

**"Highlight what changed"**: when a slider moves, the line of code that changed flashes.

## 5. Per-component Explorer specs

### Servo (pilot — built first)

- **Visual**: top-down SVG of Tower-Pro-style servo. Horn rotates live with slider. Ghost arm shows target angle. 0–180° angular ruler. Drag calibration limit pins on the ruler. Sweep mode = motion blur trail. Subtle gear-tick sound.
- **Controls**: angle 0–180, presets (0/45/90/135/180), Sweep mode + speed, Smooth-move steps/ms.
- **Calibration**: find center, set user min/max, save per-port (S1/S2) to localStorage, keyed by BLE device ID.
- **Verb**: `SRV:i,a` (i ∈ 1,2; a ∈ 0..180).

### Simple LED

- **Visual**: photorealistic 5mm LED dome. OFF = dim glass. ON = warm-up ramp + bloom corona + photon particles.
- **Controls**: ON/OFF toggle, blink Hz, pulse, fade pattern.
- **Inset**: mini Maqueen board showing which LED is lit.
- **Verb**: `LED:i,s` (i ∈ 0,1; s ∈ 0,1).

### RGB (4 px)

- **Visual**: top-down render of Maqueen board with 4 NeoPixels in real positions. Click any pixel → custom HSV wheel. Brightness drives bloom radius.
- **Controls**: per-pixel color, all-same toggle, brightness, animations (rainbow / chase / breathe / fade) with speed.
- **Verb**: `RGB:i,r,g,b` (i ∈ 0..3 or `*` for all).

### Motor

- **Visual**: robot top-down with wheel SVGs. Spokes spin at rate ∝ speed. Reverse = spin reverses. Motion lines for high speeds.
- **Controls**: speed −255..+255, direction, brake, ramp speed.
- **Calibration**: per-wheel trim (offset to drive straight), saved.
- **Verb**: `M:L,R` (signed −255..255 each).

### Buzzer

- **Visual**: speaker SVG with concentric sound waves (radius = volume, hue = freq). Piano keyboard along bottom. Web Audio plays the same tone in browser.
- **Controls**: frequency, duration, preset notes, melody picker.
- **Verb**: `BUZZ:f,ms`.

### Ultrasonic

- **Visual**: sonar sweep arc from front of robot. Distance dot pinned on arc. Polar grid (10–100 cm). Trail of last 50 readings as fading dots. Configurable green/yellow/red threshold bands. Chunky readout.
- **Controls**: polling rate, manual ping button, threshold zones.
- **Calibration**: zero offset.
- **Verb**: `DIST?` → `DIST:cm`.

### Line sensors

- **Visual**: robot bottom view with 2 "eyes." Below: draggable black tape strip kid can move under the eyes; real readings overlay simulation.
- **Controls**: polling rate, transition log.
- **Verb**: `LINE?` → `LINE:l,r`.

### IR remote

- **Visual**: photo-realistic SVG remote. Last-pressed key flashes + glows.
- **Controls**: raw-code mode / named-key mode. Learning mode: kid presses physical key, names it, app maps the code.
- **Verb**: push event `IR:code` (or poll `IR?`).

### Drive Explorer (composite — uses motors + accel telemetry)

- **Visual**: joystick + STOP button + ghost-robot canvas (dead-reckoning trail).
- **Telemetry**: live 3-axis accel graph with vertical command markers on the timeline.
- **Verb**: `M:L,R` + auto stream `ACC:x,y,z` ~20 Hz.

## 7. Telemetry overlay (the pedagogical gold)

In Drive Explorer, bottom panel = live 3-axis accel graph with command markers.

```
Acceleration (g)
  X ━━━━━╱╲━━━━━━╱╲╲━━━━     ← forward, then turn
  Y ━━━━━━━━━━━╱──╲━━━━━     ← lateral on turn
  Z ━━━━━━━━━━━━━━━━━━━━     ← gravity baseline
        ↑           ↑
   [▶ FWD 200]  [↻ L 150]    ← command markers on timeline
```

**Features:**
- Auto-scroll timeline (10 s window, two-finger drag for history).
- Hover any point → tooltip with g-value + nearest command.
- Click a marker → highlights the curve segment after it.
- Pause / play / snapshot (PNG export) / compare-runs.

**Reusable across explorers** via `js/telemetry.js`:
- Servo: commanded angle vs time + accel if on moving arm.
- Motor: command + accel + (later) wheel encoders.
- Buzzer: actual mic-captured waveform.
- Ultrasonic: distance vs time + manual-ping markers.

## 8. Add-on framework (stubbed in v0.1.0)

Reserve the verb namespace now so v0.2.0+ doesn't break the protocol:

```
ADDON:LIST                       → ADDON:none|<name>,<port>|...
ADDON:READ:<port>                → ADDON:<port>:<value>
I2C:SCAN                         → I2C:0x10,0x3C,0x76,...
I2C:READ:<addr>:<reg>:<n>        → I2C:<addr>:<reg>:<bytes>
I2C:WRITE:<addr>:<reg>:<bytes>   → ECHO
```

Firmware ships with these verbs **stubbed** (return `none` / empty). Adding real Gravity/I2C add-ons later = drop-in `js/addons/<name>.js` files, no protocol bump.
